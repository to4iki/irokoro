# テスト方針

テストは数より質を重視する。カバレッジは目的ではない。価値の低いテストは積極的に削除し、リファクタリング時に偽陽性を生む内部実装依存テストは避ける。

状態遷移やタイミングの変更は、先に失敗テストを追加する。

## フレームワーク

- Vitest + React Testing Library
- シーケンスの不変条件には fast-check（プロパティベース）を使ってよい

## 書くべきテスト

- **主要フローの最終状態**: 安全な初期値、選択の反映、一時停止で時間／シーンが止まる、自動リピートしない、など
- **境界値・ガード条件**: 期限ちょうどで終了、不正なシーケンス長、など振る舞いとして意味のある分岐
- **ドメイン不変条件**: 隣接シーンで色・形が連続しない、dwell が 6–8 秒、などプロダクト上壊れやすいルール
- **異なる入力 → 異なる出力**: デフォルト → デフォルトのようなタウトロジカルなテストは不要

## 書くべきでないテスト

- **モック呼び出し検証**: 原則、呼び出し回数だけを検証しない（要件が「呼ばれないこと」なら、呼ばれたら即失敗にする書き方は可）
- **同一観点の過剰分割**: 自然にまとまる入力・後続処理は 1 テストへ統合する
- **自明な setter / 実装詳細**: 型で守れている未知 ID の throw、oscillator 時刻など内部実装依存
